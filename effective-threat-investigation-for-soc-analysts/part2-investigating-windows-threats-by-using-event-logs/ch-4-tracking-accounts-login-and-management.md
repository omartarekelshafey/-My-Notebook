# CH 4 Tracking Accounts Login and Management

## Account login tracking

عشان تقدر تعمل Profile لحساب معين أو تتبع الحركات المشبوهة، شركة Microsoft بتوفر لك ميزة تسجيل كل  (Login attempts)، سواء كانت ناجحة أو فاشلة.

**Key Features**

* ال  Recording event logs: بيتم تسجيل تفاصيل دقيقة لكل محاولة دخول.
* ال Valuable information: السجلات دي بتشمل وقت المحاولة، اسم الحساب المستخدم، وطريقة المصادقة (Authentication method).
* ال Storage: كل سجلات المصادقة دي بتتحفظ في ملف واحد أساسي وهو الـ Security log file.

### Windows accounts

في Windows، أي  (Process) أو نشاط زي الـ Authentication  لازم يكون مرتبط بحساب محدد

**Standard Accounts**

دي الحسابات الأساسية اللي بيستخدمها الموظفين في شغلهم اليومي، ومميزاتها كالتالي:

* ال Common tasks: بتسمح للمستخدم بفتح التطبيقات، تصفح الإنترنت، والوصول للملفات المسموح له بيها.
* ال Restricted actions: الحساب ده ملوش صلاحية تثبيت برامج أو تغيير إعدادات النظام الحساسة.
* ال Security impact: استخدام النوع ده بيقلل الضرر لو الحساب اتسرق، لأن المخترق مش هيقدر يوصل لملفات النظام العميقة بدون صلاحيات أعلى.

**Default Local System Accounts**

دي حسابات مخصصة للخدمات والبرامج اللي بتشتغل في الخلفية، وساعات بتشتغل قبل ما المستخدم يعمل Login أصلاً:

* حساب SYSTEM: ده أقوى حساب في الويندوز وله سيطرة كاملة (Complete control) على كل موارد النظام

**1. حساب SYSTEM**&#x20;

* الوظيفة: ده حساب  ليه كامل الصلاحيات على ملفات النظام والنواة (Kernel).
* الهوية: بيتحكم في كل شيء محلياً، وبرا الجهاز بيظهر بهوية الكمبيوتر (Computer$).

**2. حساب NETWORK SERVICE**&#x20;

* الوظيفة: مخصص للخدمات اللي محتاجة تكلم أجهزة تانية في الشبكة "بصفة رسمية".
* الهوية: جوه جهازك اسمه NETWORK SERVICE، بس أول ما يخرج على الشبكة بيستخدم  Computer Account ($) عشان السيرفرات تعرفه.

**3. حساب LOCAL SERVICE**&#x20;

* الوظيفة: للخدمات البسيطة اللي ملهاش صلاحيات كبيرة ومحتاجة تتعامل مع الشبكة "بدون هوية".
* الهوية: محلياً صلاحياته ضعيفة، وعلى الشبكة بيدخل بصفة Null Session (مجهول)، مبيقدرش يستخدم بيانات الـ Computer$.

**4. حساب Computer Account ($)**

* الوظيفة: ده "البطاقة الشخصية" للجهاز نفسه لما ينضم للـ Domain.
* الهوية: بيعرف نفسه للـ Active Directory عشان الجهاز يقدر يوصل للموارد المشتركة.

**5. حساب ANONYMOUS LOGON**

* الوظيفة: بيسمح بالوصول لبعض المعلومات الأساسية من غير ما يطلب يوزر وباسورد صريحين.
* الهوية: مجهول تماماً (Guest/Anonymous Access).

***

### Tracking Successful Logins (Event ID 4624)

لما أي حساب (سواء كان بني آدم أو خدمة) ينجح في إنه يعمل Login، الويندوز بيسجل حدث برقم 4624. بيعرفنا مين دخل، وإزاي، ومنين.

#### Sections of Event 4624

السجل ده متقسم لـ 7 أقسام، أهمهم اللي بنستخدمه في التحقيق:

* قسم Subject: ده فيه معلومات الحساب اللي "طلب" الدخول، مش اللي دخل فعلياً. غالباً في الـ 4624 بيكون القسم ده فاضي أو ملوش لازمة أوي.
* قسم Logon Information: ده أهم حتة لأنه بيحتوي على الـ Logon Type. الرقم ده بيعرفنا المستخدم دخل إزاي (قاعد قدام الجهاز، ولا داخل ريموت، ولا خدمة...).
* قسم New Logon: ده فيه بيانات "البطل" اللي دخل فعلياً:
  * Account Name: اسم الحساب اللي دخل.
  * الAccount Domain: لو الاسم ده هو نفسه اسم الجهاز، يبقى ده Local Account، لو مختلف يبقى Domain Account.
  * ال Logon ID: رقم مميز لل (Session) عشان تقدر تتابعها بعدين وتعرف قعد قد إيه.
* قسم Process Information: بيعرفك إيه هو البرنامج أو الـ Process اللي بدأ عملية الدخول دي،
* قسم Network Information: بيعرفك الجهاز اللي الشخص داخل منه (IP واسم الجهاز). د

| **الرقم** | **النوع (Logon Type)** | **الوصف**                                              |
| --------- | ---------------------- | ------------------------------------------------------ |
| 2         | Interactive            | واحد قاعد بكرسيه قدام الجهاز وكاتب الباسورد بالكيبورد. |

### **Tracking Failed Logins (Event ID 4625)**

أي محاولة دخول "فاشلة" على الويندوز بيتم تسجيلها برقم 4625. الحدث ده هو الإنذار الأول اللي بيقولك إن فيه "هكر" بيحاول يخمن باسورودات أو يسرق حسابات.

**Failure Information Section**

ده القسم الوحيد الجديد والمهم جداً مقارنة بالحدث 4624، وفيه 3 خانات:

1. الـ Failure Reason: شرح نصي لسبب الفشل (زي "باسورد غلط").
2. الـ Status: كود برمجي (Hex) بيعبر عن حالة الفشل.
3. الـ Sub Status: وده "الأهم"، لأنه بيديك السبب الدقيق جداً للفشل بكود محدد.

**Key Failure Codes (Sub Status)**

| **الكود**  | **المعنى التقني**        |  **الوصف**                                    |
| ---------- | ------------------------ | --------------------------------------------- |
| 0xC0000064 | Username does not exist  | المخترق بيجرب أسماء مستخدمين مش موجودة أصلاً. |
| 0xC000006A | Bad password             | الاسم صح بس الباسورد غلط (محاولة تخمين).      |
| 0xC0000072 | Disabled account         | حد بيحاول يدخل بحساب مقفول (مريب جداً).       |
| 0xC0000234 | Account has been locked  | الحساب اتجمد من كتر المحاولات الغلط.          |
| 0xC000006F | Outside authorized hours | موظف بيحاول يدخل في وقت مش مسموح له بيه.      |

**Detecting Attacks (Brute Force vs. Password Spraying)**

الـ Event 4625 بيخليك تفرق بين نوعين من الهجمات:

* عملية Brute-forcing: هتلاقي محاولات فشل كتيييير جداً جاية على حساب واحد بس.
* عملية Password Spraying: هتلاقي محاولات فشل جاية من جهاز واحد بس موجهة لـ حسابات كتير مختلفة (بيجرب باسورد واحد سهل على الكل).

**Account Lockout (Event ID 4740)**

لما عدد محاولات الفشل يزيد عن الحد المسموح به في سياسة الشركة، الويندوز بيقفل الحساب وبيطلع الحدث رقم 4740.

* خانة Subject: بتبين لك مين "الجهاز" أو "الحساب" اللي اتسبب في قفل الحساب ده (زي جهاز `WIN-SOC2$`).
* خانة Account That Was Locked: اسم الحساب اللي "اتقفل" (زي حساب `Ali`).

### Tracking Successful Administrator Logins

حسابات الـ Administrator هي أكتر حسابات حساسة لازم محلل الـ SOC يراقبها. ليه؟ لأن المخترق مستحيل يحقق أهدافه (تشفير ملفات، سرقة بيانات، أو زرع برمجيات خبيثة) من غير ما يوصل لصلاحيات الأدمن.

**Event ID 4672: Special Privileges Assigned**

لما حساب له صلاحيات "خاصة" أو "إدارية" ينجح في الدخول، الويندوز بيسجل event برقم 4672.&#x20;

**Sections of Event 4672**

* قسم Account Information: فيه بيانات الحساب اللي دخل (Account Name، Account Domain، والـ Logon ID).
* قسم Privileges: وده الجزء الأهم، لأنه بيسرد لك قائمة بكل الصلاحيات الإدارية "الخاصة" اللي اتمنحت للجلسة دي (زي الحق في تعديل ملفات النظام أو التحكم في الحسابات التانية).

<figure><img src="../../.gitbook/assets/image (2).png" alt="" width="563"><figcaption></figcaption></figure>

**The "Double Event" Rule**

دي معلومة ذهبية لازم تكون عارفها؛ لما Admin يدخل على الجهاز، الويندوز مش بيطلع حدث واحد، بيطلع حدثين مع بعض في نفس اللحظة:

1. الـ Event ID 4624: وده بيقولك "فيه حساب دخل بنجاح" (زي ما شرحنا قبل كدة).
2. الـ Event ID 4672: وده بيجي وراه فوراً عشان يقولك "الحساب اللي لسه داخل ده معاه صلاحيات Admin وده كشف بصلاحياته".

#### مثال عملي من الكلام&#x20;

الحساب اللي اسمه `pbeesly` دخل على الجهاز.

* التحليل: السيستم سجل 4672، وده معناه إن `pbeesly` ده مش مجرد موظف، ده شخص معاه Special Privileges. السجل بيفصل لنا كل الصلاحيات اللي يقدر يستخدمها خلال الجلسة دي.

## Login Validation Events

تعتبر الـ Login Validation Events هي الـ events الخاصة بنتائج الـ Credential validation. والفرق الجوهري هنا هو "مكان تسجيل الـ event"؛ فبينما الـ Logon events الشهيرة زي 4624 و 4625 و 4672 بتتسجل على جهاز الموظف (Workstation) اللي بيتحاول يدخل عليه، نجد إن الـ Login Validation Events بتتسجل فقط على "السيرفر المسؤول عن عملية المصادقة".

في حالة الـ Domain account authentications، بيكون الـ Domain Controller هو المسؤول، وبالتالي هو اللي بيسجل الـ Login Validation Events. أما في حالة الـ Local account authentications، فجهاز المستخدم هو اللي بيقوم بالمهمة دي باستخدام قاعدة بيانات الـ SAM المحلية، وبيسجل الـ events دي عنده.

بتعتمد مايكروسوفت في تسجيل الـ events دي على بروتوكول المصادقة المستخدم، واللي غالباً بيكون إما NTLM أو Kerberos

***

### NTLM Protocol: Event ID 4776

لما بنستخدم بروتوكول NTLM في المصادقة، الويندوز بيسجل event برقم 4776.&#x20;

**Key Fields in Event 4776**

* خانة Logon Account: اسم الحساب اللي بيحاول يدخل.
* خانة Source Workstation: اسم الجهاز اللي المستخدم قاعد عليه وبيبعت منه الباسورد.
* خانة Error Code: لو المحاولة فشلت، الرقم اللي هنا هو اللي هيقولك السبب.

**Error Codes Correlation**

&#x20;الـ Error Codes في الحدث 4776 هي هي نفس أكواد الـ Sub Status اللي  في الـ Event 4625.

* مثال عملي&#x20;

### Login Validation Events (Kerberos Protocol)

بروتوكول Kerberos هو الأساس في أي Active Directory. بدل ما يبعت الباسورد كل شوية، هو بيستخدم نظام تذاكر. الـ events دي بتتسجل "فقط" على الـ Domain Controller لأن هو اللي بيوزع التذاكر دي.

**1. Event ID 4768: TGT Request**

* الوظيفة: الـ event ده بيقول إن المستخدم نجح في أول خطوة (المصادقة الأولية) والـ DC طلعلة تذكرة اسمها Ticket Granting Ticket (TGT).
* المعنى: المستخدم ده "معاه كارنيه" يلف بيه في الدومين لفترة معينة.

**2. Event ID 4769: Service Ticket Request**

* الوظيفة: لما المستخدم اللي معاه TGT يحب يدخل على "سيرفر ملفات" أو "قاعدة بيانات"، الـ DC بيديله تذكرة خاصة بالخدمة دي (Service Ticket).
* المعنى: ده سجل دخولك الفعلي لمورد معين (Resource) جوه الشبكة.

**3. Event ID 4771: Kerberos Pre-authentication Failed**

* الوظيفة: ده "جرس الإنذار". بيسجل فشل عملية المصادقة الأولية، يعني الـ DC رفض يديك الـ TGT أصلاً.
* المعنى: البيانات اللي اتقدمت غلط، ومفيش تذاكر هتطلع.

***

### Failure Codes in Event 4771

خلي بالك، الأكواد هنا مختلفة عن الـ events اللي فاتت (4625 و 4776)،&#x20;

| **الكود (Failure Code)** | **المعنى التقني**       | **الوصف**                                                       |
| ------------------------ | ----------------------- | --------------------------------------------------------------- |
| 0x6                      | User doesn't exist      | اليوزر اللي بيحاول يدخل مش موجود في الدومين.                    |
| 0x12                     | Password expired        | الباسورد صلاحيته انتهت ولازم يتغير.                             |
| 0x17                     | Wrong password          | الباسورد اللي اتكتب غلط.                                        |
| 0x18                     | Account disabled/locked | الحساب مقفول أو محظور حالياً.                                   |
| 0x25                     | Clock out of sync       | وقت الجهاز بعيد عن وقت الـ DC (ودي مشكلة مشهورة في الكيربيروس). |

#### مثال

الحساب `Mostafa.yahia` حاول يدخل من جهاز الـ IP بتاعه `10.0.0.5`.

* النتيجة: السيستم سجل Event 4771.
* التحليل: لازم تبص على الـ Failure Code؛ لو لقيته 0x17 يبقى مصطفى بيكتب باسورده غلط، لو 0x6 يبقى فيه حد بيخمن أسماء موظفين من الجهاز ده.
